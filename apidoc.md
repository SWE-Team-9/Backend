Base Routes: /api/v1/auth — user authentication /api/v1/oauth — third-party OAuth provider 

This module handles: 

User registration 

Login with email/password 

Google OAuth login 

Email verification 

Password recovery 

Email changes 

Session management 

JWT token lifecycle 

OAuth2 provider flow for third-party API access 

All token delivery uses httpOnly secure cookies. No tokens are ever exposed in response bodies for user authentication. Rate limiting is applied to all sensitive authentication routes via @nestjs/throttler. 

 

1. Security Architecture 

Password Hashing 

argon2 PHC competition winner, stronger than bcrypt against GPU/ASIC attacks. 

Token Strategy 

JWT access tokens: 15 minutes 

Opaque refresh tokens: 7 days 

Tokens are delivered only in httpOnly, Secure, SameSite=Strict cookies 

Refresh Token Rotation 

Every refresh request: 

issues a new refresh token 

invalidates the old one 

If a previously consumed refresh token is reused, this is treated as token theft and all user sessions are revoked immediately. 

CAPTCHA 

Google reCAPTCHA v2/v3 is required on registration to reduce automated abuse. 

Rate Limits Per Endpoint 

POST /register → 3 requests/minute per IP 

POST /login → 5 requests/minute per IP, 10/minute per email 

POST /forgot-password → 3 requests/hour per email 

POST /resend-verification → 3 requests/hour per email 

POST /request-email-change → 3 requests/hour per user 

HTTP Security 

Helmet for secure headers 

CORS restricted to frontend origin only 

Cookie Specification 

Set automatically by the backend on every authenticated response: 

Cookie: access_token 

Value: Signed JWT (HS256) 

HttpOnly: Yes 

Secure: Yes 

SameSite: Strict 

Max-Age: 15 minutes 

Cookie: refresh_token 

Value: Opaque token (hashed in DB) 

HttpOnly: Yes 

Secure: Yes 

SameSite: Strict 

Max-Age: 7 days 30 days if remember_me = true 

JWT Access Token Payload 

{ 
  "sub": "user-uuid", 
  "role": "USER", 
  "iat": 1709800000, 
  "exp": 1709800900 
}

Endpoint 6: Login with Google (OAuth 2.0) — Step 2: Callback 

Purpose Google redirects back here after the user approves access. The backend exchanges the authorization code, reads profile info, and either: 

logs in an existing user 

links an existing account 

creates a new account 

Method GET 

Endpoint /api/v1/auth/google/callback 

Authentication Required No Handled by Passport strategy. 

Query Parameters 

code — Authorization code from Google state — CSRF token that must match the original one 

Internal Behavior 

Passport exchanges the authorization code for a Google access token 

Backend reads: 

email 

display name 

avatar URL 

Three cases: 

Case A AUTH_IDENTITY exists with: 

provider = GOOGLE 

matching provider_user_id → Log user in 

Case B No Google identity exists, but a USER with the same email exists → Link Google identity to that account and log in 

Case C No account exists → Create new user, mark verified, create profile, create AUTH_IDENTITY, and log in 

Success Behavior 

Set: 

access_token cookie 

refresh_token cookie 

Then redirect to: 

{FRONTEND_URL}/auth/callback?success=true 
  

Failure Behavior 

Redirect to: 

{FRONTEND_URL}/auth/callback?error=oauth_failed 
  

Database Operations — New User (Case C)

provider = GOOGLE

provider_email = Google email 

Insert into USER_DEVICE 

Insert into USER_SESSION

Database Operations — Existing User (Case A or B) 

Select AUTH_IDENTITY by: 

provider = GOOGLE 

provider_user_id = Google ID 

If not found: 

select USER by Google email 

insert AUTH_IDENTITY to link account 

Insert into USER_SESSION 

Set cookies and redirect 

 

Endpoint 7: Refresh Access Token 

Purpose Issues a new access token using the refresh token stored in the httpOnly cookie. Implements refresh token rotation. 

If a consumed refresh token is reused, all sessions are revoked immediately. 

Method POST 

Endpoint /api/v1/auth/refresh 

Authentication Required No This endpoint reads the refresh_token cookie, not the access token. 

Request 

No request body. The refresh token is read from the cookie automatically. 

Success Response — 200 OK 

{ 
  "message": "Token refreshed successfully" 
} 
  

Both cookies are updated: 

access_token 

refresh_token 

Possible Errors 

401 Unauthorized — INVALID_REFRESH_TOKEN Cookie missing, token invalid, or session revoked 

401 Unauthorized — TOKEN_REUSE_DETECTED A previously consumed refresh token was reused. All sessions have been revoked. 

403 Forbidden — ACCOUNT_SUSPENDED Account was suspended since last login 

Database Operations 

Read refresh token from cookie 

Select USER_SESSION where: 

refresh_token_hash = hash(cookie_token) 

revoked_at IS NULL 

expires_at > now 

If not found: 

treat as token reuse 

revoke all sessions for that user 

return 401 TOKEN_REUSE_DETECTED 

If found: 

generate new refresh token 

update USER_SESSION.refresh_token_hash 

sign new JWT access token 

set new cookies 

Why Refresh Token Rotation Matters 

If an attacker steals a refresh token and uses it before the legitimate user: 

the attacker gets new tokens 

the legitimate user later presents an already consumed token 

the server detects reuse and revokes everything 

Both attacker and legitimate user are logged out, and the legitimate user must authenticate again. This is the standard defense recommended by OAuth and IETF best practices.

Endpoint 18: OAuth2 Provider — Authorize 

Purpose Entry point for third-party apps to request access to a user account. Shows a consent screen. If the user approves, the server redirects back with an authorization code. 

Method GET 

Endpoint /api/v1/oauth/authorize 

Authentication Required Yes 

Query Parameters 

client_id — string — Required The registered third-party app ID. 

redirect_uri — string — Required Must exactly match the registered redirect URI. 

response_type — string — Required Must be "code". 

scope — string — Required Space-separated permissions, for example: 

read write 
  

state — string — Required Opaque CSRF value from the client app. Must be returned unchanged. 

code_challenge — string — Optional PKCE challenge for public clients. 

code_challenge_method — string — Optional Must be "S256" if code_challenge is present. 

Success Behavior 

Server redirects with: 

{redirect_uri}?code={authorization_code}&state={original_state} 
  

Authorization code is: 

one-time use 

valid for 60 seconds 

Failure Behavior 

{redirect_uri}?error=access_denied&state={original_state} 
  

Possible Errors 

Returned either as redirect query params or JSON if redirect URI is invalid. 

invalid_client client_id not registered or inactive 

invalid_redirect_uri Redirect URI does not match registered URI 

unsupported_response_type response_type is not "code" 

access_denied User denied authorization 

Database Operations 

Select API_CLIENT where: 

client_id = provided_client_id 

is_active = true 

Verify redirect_uri matches exactly 

On approval, insert into API_AUTH_CODE: 

client_id 

user_id 

code_hash = hash(random_code) 

code_challenge 

code_challenge_method 

expires_at = now + 60 seconds 

Redirect back with the raw code

Required

Required

Module 11: Moderation & Admin Dashboard 

Base Routes 

/api/v1/reports — user-facing report submission and appeals 

/api/v1/admin — all admin and moderator endpoints 

Module Purpose 

This module handles the full moderation lifecycle for the platform: 

users submit reports about problematic content or users 

moderators and admins review reports 

admins take enforcement actions such as warning, suspending, banning, hiding, removing, or restoring content 

the platform exposes moderation analytics and dashboard metrics 

all moderation actions are recorded in an immutable audit log 

 

1. Access Control Architecture 

Guard Stack 

Every endpoint under /api/v1/admin requires both of the following guards in sequence: 

JwtAuthGuard Verifies that the requester is authenticated using a valid access_token cookie. 

RolesGuard Verifies that the authenticated user has the required system role for the endpoint. 

If either guard fails, the request is rejected before reaching the controller. 

Role Hierarchy 

USER 

can submit reports 

can submit appeals 

MODERATOR 

can view reports 

can assign reports 

can resolve or reject reports 

can moderate tracks, playlists, and comments 

cannot suspend or ban users 

cannot access platform-wide admin analytics 

ADMIN 

full access to all admin endpoints 

can warn, suspend, ban, restore users 

can view all platform dashboard analytics 

can view the full audit log 

Role Validation Behavior 

The RolesGuard reads the role from the JWT payload (role claim). For sensitive actions, the backend must also verify the role directly from the database to prevent stale JWT privilege abuse. 

Sensitive Action Role Re-Verification 

Before executing the following actions, the backend must re-query USER.system_role from the database: 

warn user 

suspend user 

ban user 

restore user 

This prevents a demoted admin from using an unexpired JWT for destructive actions. 

Admin Assignment 

Admin and moderator roles are assigned directly in the database by a super-admin or by database seed. There is no self-service promotion endpoint. 

Rate Limiting on Admin Routes 

To reduce damage if an admin account is compromised: 

destructive admin actions: 30 requests/minute per admin 

admin read endpoints: 60 requests/minute per admin 

Destructive actions include: 

warn 

suspend 

ban 

restore 

remove/hide/restore content 

bulk moderation updates
